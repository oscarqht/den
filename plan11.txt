The user said `term.paste('\x15')` inserts an empty space character.
Why? Because `paste` in xterm.js has an option `macOptionIsMeta` and might escape control characters, or maybe `\x15` is not a valid paste string so it's stripped or converted. Or ttyd intercepts `paste` event and cleans it up.
Actually, if we look at xterm.js source, `paste` method calls `coreService.triggerDataEvent(text, true)`. The `true` parameter signifies it's pasted text. Some backends treat pasted text differently (e.g., bracketed paste mode `\e[200~...\e[201~`).
Ah! Bracketed paste mode! If the terminal has bracketed paste mode enabled (which bash and zsh often do), `term.paste('\x15')` will send `\e[200~\x15\e[201~`.
When bash receives bracketed paste, it inserts the characters literally into the command line buffer instead of executing them as control shortcuts!
So `\x15` is inserted literally into the buffer, which might show up as a space or `^U` block instead of clearing the line.

To fix this, we need to send the data as a regular keypress, not as a paste.
How to send data as a keypress in xterm.js?
In xterm.js, you can trigger `onData` by using `term._core.coreService.triggerDataEvent('\x15')`.
Wait, is `triggerDataEvent` available on `term._core.coreService`? Yes, typically `term._core.coreService.triggerDataEvent(data)`.
Or maybe we can just dispatch a keyboard event to the textarea?
If we do:
```typescript
const textarea = iframe.contentDocument?.querySelector("textarea.xterm-helper-textarea");
if (textarea) {
    textarea.dispatchEvent(new KeyboardEvent('keydown', {
        bubbles: true, cancelable: true, key: 'u', ctrlKey: true
    }));
}
```
If we dispatch a real `keydown` event on the textarea, xterm.js will handle it and send the right bytes.
Wait, we are inside `attachCustomKeyEventHandler`. The `event` parameter is the original `KeyboardEvent` (which is `Cmd+Backspace`). We can't just dispatch a new `KeyboardEvent` while handling this one because it might get recursive or just be easier to send the data directly.
But wait! What if we just do: `term._core?.coreService?.triggerDataEvent('\x15')`?
Let's see if we can find how `term` is used elsewhere in SessionView.

Ah! They use `textarea.dispatchEvent` for `Enter`!
Why did they do that? Probably because `term.paste('\r')` in bracketed paste mode would just paste a literal carriage return, instead of executing the command!
This confirms my bracketed paste theory!

So to simulate `Ctrl+U`, we should either:
1. Dispatch a `KeyboardEvent` for `Ctrl+U`.
2. Or use `term.paste` but somehow bypass bracketed paste (not possible via public API).
3. Or `term._core.coreService.triggerDataEvent('\x15')`.

Let's try to dispatch `Ctrl+U`.
```typescript
                            if (event.type === 'keydown' && event.metaKey && (event.key === 'Backspace' || event.key === 'Delete')) {
                                const textarea = iframe.contentDocument?.querySelector("textarea.xterm-helper-textarea");
                                if (textarea) {
                                    textarea.dispatchEvent(new KeyboardEvent('keydown', {
                                        bubbles: true,
                                        cancelable: true,
                                        key: 'u',
                                        ctrlKey: true,
                                        view: win
                                    }));
                                } else if (term._core?.coreService?.triggerDataEvent) {
                                    term._core.coreService.triggerDataEvent('\x15');
                                } else {
                                    // fallback (might insert space if bracketed paste is on)
                                    term.paste('\x15');
                                }
                                return false;
                            }
```
Wait, if we dispatch `keydown`, xterm.js listens to `keydown` on the textarea.
Does xterm.js process `keydown` with `ctrlKey: true` and `key: 'u'` correctly? Yes, it translates it to `\x15`.
Let's verify what `KeyboardEvent` properties xterm.js uses for `Ctrl+U`. It uses `ctrlKey` and `key: 'u'`, or `keyCode: 85`.
